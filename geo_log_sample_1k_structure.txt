Описание структуры geo_log_sample_1k.jsonl
=========================================

Формат файла
------------
Файл имеет формат JSONL: каждая строка является отдельным JSON-объектом.
В выборке 1000 строк / записей.

Верхний уровень записи
----------------------
Каждая запись содержит одинаковый набор полей:

1. latitude
   Тип: number
   Количество заполненных записей: 1000 / 1000
   Описание: широта пользовательской геопозиции.
   Пример: 37.874904

2. longitude
   Тип: number
   Количество заполненных записей: 1000 / 1000
   Описание: долгота пользовательской геопозиции.
   Пример: 32.492129

3. message_id
   Тип: string
   Количество заполненных записей: 1000 / 1000
   Описание: UUID сообщения пользователя или связанного запроса.
   Пример: "7475f82f-d35f-411c-876b-a233f6cd9d3c"

4. model_response_full
   Тип: array
   Количество заполненных записей: 1000 / 1000
   Описание: полный ответ модели и служебный контекст. Это самое сложное поле:
   массив содержит вложенные массивы, null-значения, словари сообщений и строки,
   внутри которых дополнительно закодирован JSON.

5. model_response_timestamp
   Тип: integer
   Количество заполненных записей: 1000 / 1000
   Описание: Unix timestamp ответа модели в секундах.
   Пример: 1771686554

6. prod_classifier_agent
   Тип: string
   Количество заполненных записей: 1000 / 1000
   Описание: продукционный агент / классификатор, который обработал запрос.
   Пример: "Geo Agent"

7. query
   Тип: string
   Количество заполненных записей: 1000 / 1000
   Описание: исходный пользовательский запрос.
   Пример: "Konumumdan en yakın rotayı çıkar "

8. region_id
   Тип: integer
   Количество заполненных записей: 1000 / 1000
   Описание: идентификатор региона пользователя или региона поиска.
   Пример: 101474

9. region_name
   Тип: string
   Количество заполненных записей: 1000 / 1000
   Описание: человекочитаемое название региона.
   Пример: "Konya"

Структура model_response_full
-----------------------------
Поле model_response_full во всех записях является массивом. В наблюдаемой выборке
массив обычно состоит из 9 позиций с такими типами:

model_response_full[0]
  Типы: null в 898 записях, array в 102 записях.
  Назначение неочевидно по выборке; вероятно, опциональное служебное поле.

model_response_full[1]
  Тип: array в 1000 записях.
  Обычно содержит один строковый UUID.

model_response_full[2]
  Тип: array в 1000 записях.
  Обычно содержит один UUID, часто совпадающий с message_id.

model_response_full[3]
  Тип: array в 1000 записях.
  Обычно содержит строку с вариантом / профилем ответа.
  Пример: ["default"]

model_response_full[4]
  Тип: array в 1000 записях.
  Обычно содержит одну строку, которая сама является JSON-объектом.
  Эту строку нужно дополнительно разбирать через json.loads.

  Частые ключи внутри распарсенной JSON-строки:
  - IsCompleteResults
  - IsFinalMarkdownText
  - ResponseStatus
  - StructuredData
  - TargetMarkdownText
  - ResponseMessageId
  - RetryRecommendationMs
  - RephraseItems
  - LinksData

model_response_full[5]
  Тип: array в 1000 записях.
  Содержит вложенный массив. В части записей может быть пустым.

model_response_full[6]
  Типы: null в 864 записях, array в 136 записях.
  Опциональное служебное поле.

model_response_full[7]
  Тип: null в 1000 записях.
  В текущей выборке всегда пустое.

model_response_full[8]
  Типы: array в 998 записях, null в 2 записях.
  Обычно содержит историю диалога / трассировку выполнения:
  - сообщения пользователя;
  - сообщения ассистента;
  - вызовы инструментов;
  - ответы инструментов;
  - дополнительные поля extra_info, next_start_links_v2, next_start_media_v2.

Типовая структура элемента истории диалога
------------------------------------------
Внутри model_response_full[8] встречаются объекты сообщений со следующими полями:

content
  Тип: string
  Описание: текст сообщения, ответ инструмента или JSON-строка.
  В ряде случаев content содержит строку, которая является JSON-объектом или JSON-массивом.

role
  Тип: string
  Описание: роль сообщения.
  Частые значения: "user", "assistant", "tool".

extra_info
  Тип: object
  Описание: служебная информация о сообщении.
  Примеры ключей: timestamp_seconds, current_agent.

tool_calls
  Тип: array или null
  Описание: список вызовов инструментов ассистентом.
  Типовые вложенные поля:
  - id
  - function.name
  - function.arguments

tool_call_id
  Тип: string
  Описание: идентификатор вызова инструмента, к которому относится tool-сообщение.

next_start_links_v2
  Тип: object
  Описание: ссылки, связанные с ответом. Может содержать links_items и links_descriptions.

next_start_media_v2
  Тип: array
  Описание: медиа-блоки / карточки организаций или геообъектов.

JSON-строки внутри JSON
-----------------------
В файле есть поля, которые выглядят как строки, но фактически содержат JSON.
Их нужно разбирать отдельно.

Основные места:

1. model_response_full[4][0]
   Всего: 1000 случаев.
   Это JSON-объект с финальным ответом модели, ссылками, структурированными блоками
   и статусом ответа.

2. model_response_full[8][...].content
   Всего найдено: 3274 JSON-строки в content.
   Возможное содержимое:
   - GEO_API_response
   - WEB_SEARCH_API_response
   - Error
   - Status
   - JSON-массивы с результатами поиска

3. tool_calls[].function.arguments
   Тип: string
   Часто содержит JSON с аргументами вызова инструмента.
   Пример: {"query":"rota"}

Практические рекомендации по парсингу
-------------------------------------
1. Читать файл построчно как JSONL.
2. Для каждой строки выполнить json.loads(line).
3. Верхние поля latitude, longitude, message_id, timestamp, query, region_id,
   region_name можно использовать напрямую.
4. Для model_response_full[4][0] выполнить дополнительный json.loads, если строка
   начинается с "{" или "[".
5. В истории model_response_full[8] рекурсивно проверять строковые поля content и
   function.arguments: если строка начинается с "{" или "[", пробовать json.loads.
6. Учитывать null-значения в позициях model_response_full[0], [6], [7], [8].
7. Не считать структуру model_response_full полноценной схемой с именованными
   полями: это позиционный массив, поэтому при обработке лучше проверять длину
   массива и тип каждого элемента.
